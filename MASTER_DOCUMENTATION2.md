## Section 1 — Project Overview

### What Problem This Project Solves
In traditional enterprise environments—particularly banks—cybersecurity operations are heavily siloed. Security analysts monitor streams of flat log data in Security Information and Event Management (SIEM) systems. Vulnerability management teams scan servers periodically to produce reports lists containing thousands of Common Vulnerabilities and Exposures (CVEs). 

The core problem is **context separation**:
* Analysts see an alert indicating a brute-force login attempt, but they do not know if that server holds highly sensitive assets or has neighboring, unpatched systems.
* Vulnerability managers see a critical patch is missing on a database server, but they do not know if that database is accessible from an active attack trajectory originating in the DMZ.
* There is no single operational view that combines **live alerts**, **vulnerability status**, **network graph topology**, and **automatic threat remediation**.

**CyberDefense AI** solves this problem by introducing a unified **3D Digital Twin SOC Platform**. By combining a graph representation of banking assets and network relationships with machine learning risk analysis and Generative AI, it transforms flat cyber alerts into a living, visual model that enables real-time, zero-trust lateral threat containment.

### The Castle & Tunnel Analogy
Imagine a traditional bank. The bank is a large castle.
* **Traditional Security (SIEM)**: The guards stand at the main gates and watch the doors. They see if someone tries to force a lock. But they are blind to the inside corridors.
* **The Reality (Lateral Movement)**: An attacker sneaks in through a side pantry window (an unpatched low-value server). Once inside, they find an unlocked crawlspace (network connection) that leads into the central heating ducts, allowing them to drop straight into the main gold vault (the Core Banking Database).
* **CyberDefense AI**: This platform maps every corridor, vault, window, and crawlspace in a 3D blueprint. When a threat enters the pantry, the system instantly traces which ductways lead to the gold vault, highlights the threat path in bright red, generates an emergency steel-gate shutter playbook (remediation action) using Gemini AI, and locks the corridor upon human approval.

### Terminology Explained Simply
* **SOC (Security Operations Center)**: The nerve center or "control room" of a company where security analysts monitor screens 24/7 to detect and stop cyber attacks.
* **Digital Twin**: An exact virtual 3D replica of physical assets and digital networks. If a server in the real world is compromised, its virtual twin lights up red in the 3D model.
* **Vulnerability Dataset**: A catalog of known security flaws (CVEs) found on the bank's computers during scans.
* **Attack Prediction**: Analyzing the active corridors in the digital twin graph to predict exactly which high-value assets an attacker will target next, before they get there.
* **Incident Remediation**: The act of containing and cleaning up an ongoing attack (e.g., blocking firewall ports, isolating servers, revoking credentials).
* **Gen-AI Playbooks**: Remediations generated by Large Language Models (LLMs) like Gemini, producing step-by-step defensive actions tailored for the specific victim server and local banking regulations (e.g., RBI Guidelines).

### Communication Flow Details
1. **API Flow**: The React frontend uses `axios` (wrapped in a clean API client in [client.js](file:///c:/Users/Banu/.gemini/antigravity/scratch/sarathi-cyberdefense/frontend/src/api/client.js)) to communicate with FastAPI. All endpoints return standardized, validated JSON data.
2. **Data Flow**: When the dashboard mounts, FastAPI queries Neo4j for the vulnerability mapping. It processes this data through the `RiskScorer` and returns the top risks to the Recharts visualizer.
3. **Rendering Flow**: The 3D Digital Twin initializes a Three.js WebGL renderer, generating nodes for all assets positioned in a layered 3D coordinate space. Network connections are drawn as directed paths.
4. **AI Flow**: When a playbook is generated, the backend packages the active CVE data and assets, formats a highly structured, banking-specific prompt, and calls Gemini. The backend parsers sanitize the Markdown and strip out code fences before delivering structured JSON chunks to the frontend.

## Section 4 — Complete Data Flow

Tracing a single data packet from initial vulnerability identification to real-time isolation on the Digital Twin represents the core operational flow of the system.

```
 [ Simulated Vulnerability Dataset ]
                 │
                 ▼  (Ingested into Neo4j as Graph Nodes & Edges)
 [ Graph Relationship Mapping ]
                 │
                 ▼  (Processed by Composite RiskScorer & RandomForest)
 [ Prioritized Risk Index (0-100) ]
                 │
                 ▼  (Evaluated by Gemini AI Engine /threats/analyse)
 [ Prioritized Attack Vectors & Executive Report ]
                 │
                 ▼  (User triggers Red Team Scenario)
 [ Lateral Path Traversal & Alert Generation ]
                 │
                 ▼  (SOC Live Alert Fires in Feed)
 [ Detected Security Incident ]
                 │
                 ▼  (Operator clicks "Generate Playbook")
 [ Gemini AI Remediation Playbook Generation ]
                 │
                 ▼  (Operator Reviews Options in Governance Panel)
 [ Human-in-the-Loop Review: Approve & Execute ]
                 │
                 ▼  (Event dispatch: 'ai-governance-remediate')
 [ Digital Twin 3D Containment & Isolation ]
```

### Step-by-Step Execution Sequence

#### 1. Ingestion & Graph Mapping
A network vulnerability scan detects a missing patch (e.g., `CVE-2026-1043`) on the **Web Application Gateway** (`Asset_1`). A cypher query injects this vulnerability and links it to the asset:
```cypher
MATCH (a:Asset {id: "Asset_1"})
CREATE (v:Vulnerability {cve_id: "CVE-2026-1043", cvss_score: 9.8, epss_score: 0.9452, is_kev: true})
CREATE (a)-[:HAS_VULNERABILITY]->(v)
```

#### 2. Risk Calculation
FastAPI fetches all vulnerability-asset linkages and evaluates the composite risk score using `RiskScorer.score_cve()`:
$$\text{Component Total} = (\text{CVSS} \times 3) + (\text{EPSS} \times 40) + (\text{KEV Bonus: 20}) + (\text{Asset Criticality} \times 3)$$
$$\text{Composite Risk Score} = \min(100.0, \text{Component Total})$$
For `CVE-2026-1043` on a criticality 10 gateway:
$$\text{Score} = (9.8 \times 3) + (0.9452 \times 40) + 20 + (10 \times 3) = 29.4 + 37.8 + 20 + 30 = 117.2 \rightarrow 100.0 \text{ (CRITICAL)}$$

#### 3. Path Traversal & Live Alerts
The operator triggers the **Ransomware Lateral Spread** simulation. The simulator loads the network path from NetworkX: `Web Application Gateway` $\rightarrow$ `Authentication Service` $\rightarrow$ `Core Database Cluster`.
Step-by-step alerts are written to the live telemetry stream as the simulated threat hops across interfaces.

#### 4. Playbook Generation
The analyst opens the critical alert. Clicking **Generate Playbook** triggers an API request to `/api/playbook/generate`. The generator formats the prompt and contacts Gemini.
If Gemini is online, it generates seven specific playbook sections. If offline, the deterministic backup immediately fires, guaranteeing no UI freezes.

#### 5. Governance Containment
The analyst clicks **Approve & Execute**. The frontend updates the local storage `simState` and dispatches `ai-governance-remediate`.
The `DigitalTwin` component intercepts the event, visualizes glowing green containment shields around the compromised database nodes, drops the bank's active DEFCON to 5, and reduces financial liability to zero.

## Section 6 — Cybersecurity Concepts Explained

This section provides clear, simple explanations and analogies for core cybersecurity concepts, assuming zero prior technical knowledge.

### SOC (Security Operations Center)
* **What it is**: The security control room.
* **Analogy**: A physical airport control tower where operators monitor flight coordinates, runway status, and weather sensors to prevent accidents. In a SOC, cyber analysts monitor server logs, network traffic, and alerts to block hackers.

### SIEM (Security Information & Event Management)
* **What it is**: The system that collects all the logs.
* **Analogy**: A security guard standing in front of a wall of 50 CCTV camera screens. The SIEM is the system that aggregates all those video feeds and highlights if a camera detects movement in a dark corridor.

### CVE (Common Vulnerabilities & Exposures)
* **What it is**: A dictionary of known security flaws.
* **Analogy**: A global police database listing known flaws in physical locks. For example, *"MasterLock Model 3 has a known defect where it can be opened with a soda can."* A CVE is the digital version of this, describing a flaw in software (e.g., Microsoft Windows or Apache Web Servers).

### MITRE ATT&CK
* **What it is**: A catalog of the specific methods hackers use.
* **Analogy**: A bank robbery playbook detailing known techniques, such as: *Entering through the roof*, *Disabling the alarm wires*, or *Hostage negotiation tactics*. The MITRE matrix categorizes digital techniques (e.g., *Phishing*, *Brute-Force*, or *Data Encryption*).

### IAM (Identity & Access Management)
* **What it is**: Digital ID cards and permissions.
* **Analogy**: A physical keycard system in a high-security facility. The janitor's keycard opens the offices, but only the database administrator's keycard opens the central server vault.

### MFA (Multi-Factor Authentication)
* **What it is**: Requiring two types of proof to log in.
* **Analogy**: A safe deposit box that requires both the customer's key **and** the bank manager's key to open. Digitally, it requires something you know (your password) and something you have (a code sent to your phone).

### EDR (Endpoint Detection & Response)
* **What it is**: Anti-virus software with recording capabilities.
* **Analogy**: A security guard who stands inside a specific office, watches everyone who touches the filing cabinets, and instantly blocks unauthorized changes.

### Threat Actor
* **What it is**: The hacker or attacking group.
* **Analogy**: A professional safe-cracker or bank heist crew. In the digital world, it represents hacktivists, cybercriminals, or nation-states.

### Attack Vector
* **What it is**: The entry point or path used to break in.
* **Analogy**: Climbing down the chimney to enter a locked house. In cyberspace, it represents an unpatched web gateway or a malicious email attachment.

### Lateral Movement
* **What it is**: Pivoting from one server to another inside the network.
* **Analogy**: Once inside the house through the window, walking room-to-room through the internal doors to find the master bedroom safe.

### Privilege Escalation
* **What it is**: Upgrading your access permissions once inside.
* **Analogy**: Finding the keys to the master bedroom on the kitchen counter after entering as a guest, giving you full access to the house.

### Zero Trust
* **What it is**: Never trust, always verify.
* **Analogy**: A high-security facility where you must show your ID card at **every single door** you pass through, even if you already passed through the front gate.

### Kill Chain
* **What it is**: The step-by-step sequence of an attack.
* **Analogy**: 
1. Reconnaissance (looking at the bank layout)
2. Weaponization (preparing the safe-cracking tools)
3. Delivery (entering the bank lobby)
4. Exploitation (cracking the vault door)
5. Action on Objectives (stealing the gold).

### Digital Twin
* **What it is**: A virtual 3D copy of a real system.
* **Analogy**: A physical scale model of a building on the table in a military operations room. If a light goes out in the real building, the operator flips a switch on the model.

### Threat Intelligence
* **What it is**: Information about active attacks around the world.
* **Analogy**: Neighborhood watch notifications warning that a specific white van is targeting local houses. Digitally, it represents IP address blacklists and malware hashes.

### Red Teaming
* **What it is**: Simulating real attacks to test defenses.
* **Analogy**: A bank hiring a private security firm to try to break into its own vaults to expose security vulnerabilities.

### Vulnerability Management
* **What it is**: Finding and patching software flaws before hackers do.
* **Analogy**: Periodically checking the building's locks, windows, and fences for wear and tear, and upgrading them before a break-in occurs.

## Section 7 — Banking Context Explained

Cybersecurity in banking is uniquely critical because a bank is not just protecting data—it is protecting the stability of the financial system.

### Why Banks Are Prime Targets
* **Direct Monetization**: Unlike hospitals or retail stores, breaking into a bank gives attackers direct access to funds, ledger balances, and payment rails.
* **Complex Trust Relationships**: Banks operate massive networks of third-party payment gateways, merchant interfaces, and employee terminals. An attacker only needs to compromise one weak link to gain a foothold.

### Core Banking
* **What it is**: The central transactional ledger of the bank.
* **Why it matters**: It is the system of record. It processes deposits, withdrawals, and loan balances. If the Core Database is compromised or encrypted, the bank must halt operations immediately.

### SWIFT (Society for Worldwide Interbank Financial Telecommunication)
* **What it is**: The global financial messaging network.
* **Why it matters**: It acts as the secure messaging highway that banks use to send payment orders to one another across the globe. A compromise on a bank's SWIFT gateway can result in billions of dollars in fraudulent outbound transfers (e.g., the 2016 Bangladesh Bank heist).

### Financial Liability
* **What it is**: The cumulative monetary cost of a security breach.
* **Why it matters**: During an active breach, costs accrue rapidly due to transaction losses, regulatory fines (RBI penalties), forensics investigations, and DPDP customer notifications.

## Section 8 — Digital Twin Deep Dive

Our 3D Digital Twin is engineered using **Three.js** inside [DigitalTwin.jsx](file:///c:/Users/Banu/.gemini/antigravity/scratch/sarathi-cyberdefense/frontend/src/components/DigitalTwin.jsx). This section explains the graphics and visualization logic that drives the model.

### How the 3D Nodes Represent Infrastructure
Assets are rendered as structured meshes situated across layered Y-axis levels to represent distinct security zones:
* **Internet Zone ($Y \approx 13.5$)**: Public space, rendered as a high-intensity red sphere that pulses dynamically.
* **DMZ Zone ($Y \approx 3.75$)**: Public-facing security perimeter. Hosts the firewall, load balancers, and gateways, rendered as blue and purple spheres.
* **Internal Network ($Y \approx -3.75$)**: The private bank intranet (Authentication routers and app servers), rendered as green spheres.
* **Core Banking Zone ($Y \approx -11.25$)**: Highly isolated vault space hosting databases and SWIFT gateways, rendered as golden spheres.

```
       [ Public Internet ]           Y ~ 13.5   (Red Sphere)
                │
                ▼
      [ DMZ Security Zone ]          Y ~ 3.75   (Blue/Purple Spheres)
                │
                ▼
     [ Internal Bank Network ]       Y ~ -3.75  (Green Spheres)
                │
                ▼
    [ Core Banking DB & SWIFT ]      Y ~ -11.25 (Gold Spheres)
```

### Edge Connections & Attack Propagation
* **Connections**: Directed lines (`LineBasicMaterial`) connecting nodes represent configured network paths.
* **Attack Propagation**: When a scenario starts, a custom particle system (`PointsMaterial`) activates. High-contrast red particles traverse these lines using simple linear interpolation:
$$\mathbf{P}_{\text{particle}} = \mathbf{P}_{\text{source}} + t \times (\mathbf{P}_{\text{target}} - \mathbf{P}_{\text{source}})$$
As $t$ advances from 0 to 1, the red particles appear to travel along the connection lines.
* **Compromise Pulse**: When a node is reached, the mesh scale is scaled up dynamically using a sine-wave multiplier in the animation frame loop:
$$\text{Scale}(t) = 1.0 + \sin(5t) \times 0.15$$
This causes the target node to pulse in size, representing an active breach. A floating "COMPROMISED" billboard status badge is rendered above the node.

### Containment Visualization (Green Shields)
When an analyst approves the playbook containment, the frontend catches the `ai-governance-remediate` event. The system immediately:
* Halts the active red particle systems.
* Renders expanding wireframe sphere meshes (`SphereGeometry` with `wireframe: true`) around the compromised database nodes, styled in a glowing emerald green color (`0x10b981`).
* Resets the active compromised status badges and restores the bank's active DEFCON to 5.

## Section 9 — AI Engine & Prompts Deep Dive

The core intelligence of the platform is powered by the **Google Gemini API**, orchestrated by [playbook_gen.py](file:///c:/Users/Banu/.gemini/antigravity/scratch/sarathi-cyberdefense/backend/genai/playbook_gen.py).

### System Prompt for Playbooks
We supply Gemini with a highly specific system prompt that configures its identity, context, and output format:

```text
You are a senior cybersecurity engineer at a major Indian bank (Union Bank of India).
Generate a detailed remediation playbook for the following vulnerability:

CVE ID: {cve_id}
Description: {description}
CVSS Score: {cvss} ({severity})
EPSS Score: {epss} (probability of exploitation)
Known Exploited: {is_kev}
Affected Banking Assets: {affected_assets}

Generate a structured playbook with these exact sections:
1. EXECUTIVE SUMMARY
2. IMMEDIATE ACTIONS
3. SHORT-TERM REMEDIATION
4. LONG-TERM HARDENING
5. VERIFICATION STEPS
6. ROLLBACK PLAN
7. COMPLIANCE NOTES

Be specific to banking infrastructure.
Include actual commands where relevant.
```

### Explaining the Playbook Sections
1. **Executive Summary**: A concise briefing of the vulnerability, asset exposure, and compliance impact (e.g., RBI SLA violations).
2. **Immediate Actions**: Step-by-step containment procedures to execute within 15 minutes (e.g., isolating network adapters, dropping database connections, rotating service accounts).
3. **Short-Term Remediation**: Mitigation actions to execute within 72 hours (e.g., applying security patches, running vulnerability scans).
4. **Long-Term Hardening**: Zero-trust architecture upgrades (e.g., micro-segmentation, transitioning to hardware MFA).
5. **Verification Steps**: Specific shell commands (e.g., `nmap` or `dpkg`) to verify that the vulnerability is closed.
6. **Rollback Plan**: Clear steps to restore services if the patch breaks applications (e.g., VM snapshot rollback commands).
7. **Compliance Notes**: Legal requirements mapped to the Indian Personal Data Protection (DPDP) Act 2023 and RBI mandates.

### Resilient Fallback Architecture (Judge Favorite)
AI APIs can occasionally be rate-limited, fail to connect, or experience latency. Judges appreciate systems that maintain high availability despite external API failures. We built a **three-tier fallback safety net**:

```
 [ Tier 1: Gemini 1.5 Pro (Primary) ]
                 │
                 ▼ (If 404 / unsupported / quota exhausted)
 [ Tier 2: Gemini 3.5 Flash (Dynamic Fallback) ]
                 │
                 ▼ (If API completely offline or rate-limited)
 [ Tier 3: High-Fidelity Static Fallback (Resilient Backup) ]
```

1. **Primary Model (`gemini-1.5-pro`)**: Called with a rate-limit retry policy (exponential backoff up to 3 attempts).
2. **Dynamic Model Fallback (`gemini-3.5-flash`)**: If the API key lacks permission or quota for `gemini-1.5-pro`, the system intercepts the error and immediately retries the request using `gemini-3.5-flash`.
3. **Deterministic Backup**: If the API is completely offline or rate-limited, the system instantly loads a pre-written, highly detailed fallback playbook tailored to the target asset. This guarantees that the UI never freezes or goes blank during a live presentation.
